# MCP Chase: Joker Protocol

**Course:** Orchestration of AI Agents · **Assignment:** EX06 — Dual AI Agent
Conversation via MCP Servers

A dual autonomous AI-agent pursuit game. A **Cop** and a **Thief**, each backed
by its **own MCP-shaped tool server**, converse in **free natural language** and
chase each other on a grid under **partial observation**. The graded point of
the assignment is **orchestration** — wiring up two autonomous agents that
understand each other and act — not winning the game or strategy quality.

> **Status:** Phases 0–6 complete; **Phase 7 (submission polish)** in progress.
> The default run is **fully offline and deterministic** for reproducibility.
> Optional local **Ollama** reasoning and optional real **Gmail** delivery are
> implemented and gated behind explicit flags. **No GUI** (intentionally out of
> scope — see *Known Notes / Honest Limitations*).

---

## Quick Start

```bash
# 1. (optional) create a virtual environment, then install deps
pip install -r requirements.txt

# 2. run the full local simulation (6 sub-games, deterministic, offline)
python -m src.main

# 3. run the tests
python -m pytest

# 4. build the email from the latest report WITHOUT sending (dry-run, default)
python -m src.gmail.cli --dry-run
```

`python -m src.main` needs **no third-party packages** — it runs on the Python
standard library alone. The extras in `requirements.txt` (FastMCP, Google API
client) are only needed for the optional MCP-transport and real-Gmail paths.

### How to run the simulation

```bash
python -m src.main                       # uses ./config.json
python -m src.main --config config.json  # explicit config path
```

Runs all 6 sub-games through the MCP-shaped tool layer and writes the log and
JSON report (see *Outputs*). Deterministic mode is the default (`random_seed`
in `config.json`), so the result is reproducible: **Cop 120 / Thief 30**.

### How to run the tests

```bash
python -m pytest          # full suite
python -m pytest -q       # quiet
```

Expected: **52 passed, 1 skipped**. The one skip is `test_mcp_servers.py`'s live
FastMCP construction, which is skipped automatically when the optional `mcp` /
`fastmcp` package is not installed.

### How to run the Gmail dry-run

```bash
python -m src.gmail.cli --dry-run   # or just: python -m src.gmail.cli
```

Loads `results/reports/final_report.json`, validates it is JSON, builds the
email subject/body/attachment, prints a summary, and writes
`results/reports/email_dry_run.json`. **It never contacts the Gmail API.**

### How to optionally use Ollama (real local LLM reasoning)

Deterministic reasoning is the default. To use a real local model instead:

```bash
# one-time: start Ollama and pull the small local model
ollama serve
ollama pull smollm2:135m

# then set "agent_provider": "ollama" in config.json and run
python -m src.main
```

No API keys or secrets are used — Ollama is a **local** HTTP service at
`http://localhost:11434`. If it is not running, the run **fails clearly** with
an `OllamaUnavailable` message (`ollama serve` / `ollama pull`); output is never
faked. See *Known Notes* for the malformed-JSON fallback behavior.

## Architecture

Six cooperating layers; the **engine holds the only true state `S`**, and agents
see **only their partial observation `Ωᵢ`**.

| Layer | Modules | Responsibility |
|-------|---------|----------------|
| **Engine** (true state `S`) | `src/engine/board.py`, `rules.py`, `observation.py`, `scoring.py`, `game_loop.py` | Grid + positions + barriers, legal moves (incl. diagonal), capture, partial-view construction, scoring, and the sub-game / 6-game driver. |
| **Agents** (reasoning) | `src/agents/base_agent.py`, `cop_agent.py`, `thief_agent.py`, `provider.py`, `prompts.py` | Turn each agent's input (partial obs, last message, tool menu, objective) into a **structured decision** via a pluggable provider (deterministic or Ollama). |
| **MCP servers** (tools only) | `src/mcp/cop_server.py`, `thief_server.py`, `session.py` | Two independent FastMCP servers, one per agent, exposing **tools only** (no LLM) over a shared `GameSession` that wraps the engine. |
| **Local MCP-shaped tool layer** | `src/tools/local_adapter.py`, `dispatcher.py`, `messages.py` | In-process implementation of the **same tool vocabulary** (`cop.*` / `thief.*`) that the game loop drives directly — no network, no blocking server — and logs every call. |
| **Reporting** | `src/reporting/report_builder.py` | Assemble the structured `final_report.json` from the completed series. |
| **Gmail delivery** | `src/gmail/message_builder.py`, `sender.py`, `cli.py` | Validate the report, build the email + attachment, and either dry-run (default) or send over Gmail OAuth (`--send`). |

**Key separation:** the LLM (when enabled) lives in the **agent/client layer**;
the MCP servers **expose tools only** and never run a model. A normal
`python -m src.main` run uses the in-process tool layer and **does not start** a
blocking MCP server — the FastMCP server modules share the same tool names and
shapes and are exercised by their own tests.

## Dec-POMDP Mapping

The pursuit is modeled as a Decentralized Partially Observable Markov Decision
Process `⟨ n, S, {Aᵢ}, P, R, {Ωᵢ}, O, γ ⟩`:

| Symbol | In this project |
|--------|-----------------|
| `n` | Number of agents = **2** (Cop, Thief). |
| `S` | True state: both agents' grid positions + the set of barrier cells. |
| `Aᵢ` | Per-agent actions: move to an adjacent cell (incl. diagonal) or stay; the Cop may additionally place a barrier. |
| `P` | Transition: how a chosen action updates positions / barriers on the board. |
| `R` | Reward / scoring: Cop win → Cop 20 / Thief 5; Thief win → Cop 5 / Thief 10 (`scoring.py`). |
| `Ωᵢ` | Each agent's **partial observation** (its own position + limited view of the opponent), built by `observation.py`. |
| `O` | Observation function mapping true `S` → each agent's `Ωᵢ`. The optional Joker perturbs **only** this layer. |
| `γ` | Discount factor — reserved for the optional Q-learning extension; not used by the deterministic baseline. |

## Joker Protocol (Optional Extension)

Our optional creative extension, **off by default** (`joker_enabled: false`).
With it disabled, the project follows EX06 exactly.

- The **winner of a sub-game** receives **one Joker Card** for the **next sub-game**.
- Playing the Joker **injects one plausible false observation signal** into the
  opponent's partial observation for a single turn.
- It is a pure **observation-layer** extension — a one-shot perturbation of the
  Dec-POMDP observation function `O`.

**It never** creates a second physical Thief, teleports an agent, changes
scoring, or replaces any baseline rule. True state `S`, transitions `P`, and
rewards `R` are untouched — only the observation `Ωᵢ` is affected. A unit test
(`tests/test_observation.py`) asserts the injection never mutates `S`.

## Known Notes / Honest Limitations

- **Deterministic mode is the default** for reproducibility. `python -m src.main`
  runs fully offline with a fixed `random_seed`, so every run produces the same
  baseline totals (**Cop 120 / Thief 30**).
- **Ollama mode is supported and was smoke-tested**, but small local models
  (e.g. `smollm2:135m`) may return **malformed or illegal JSON**. When that
  happens the agent **falls back to the deterministic candidate action** and
  records the fallback in `reasoning_summary`. A *missing* Ollama server is a
  different case: it raises `OllamaUnavailable` rather than faking output.
- **Gmail real send requires local OAuth credentials** and **was not live-sent**
  in this submission. `--dry-run` is the default and validates + builds the email
  without contacting the API; a real send only happens if you run
  `python -m src.gmail.cli --send` with your own `credentials.json` / `token.json`.
- **No GUI.** A dashboard/visualization was intentionally **not implemented** —
  it is optional in the assignment and riskier to add near submission. Everything
  is observable via the terminal output, the JSONL log, and the JSON report.
- **Live MCP transport / orchestrator:** the two FastMCP server modules are
  implemented and tested for their tool surface, but a normal run drives the
  in-process tool layer (same tool names/shapes) rather than starting the servers
  over a live transport.

## Observed baseline result (produced by a real run, not invented)

With the baseline `config.json` (5×5 grid, 25 moves, `random_seed: 42`,
deterministic provider) the Cop captures the Thief in every sub-game (3–4 moves
each), giving **Cop = 120, Thief = 30**. This is the expected outcome of *simple*
deterministic policies: in discrete king-move pursuit on a bounded grid, a
pursuer that moves second corners a one-step-greedy evader. The Thief-win branch
and its `5 / 10` scoring are verified through the real game loop by
`tests/test_game_loop.py` (short-horizon sub-game).

> Note: the 30–90 score band in `prd.md` §7.4 describes a **group** that plays 3
> sub-games as Cop and 3 as Thief. This self-play run instead uses one policy as
> Cop and one as Thief for all 6 sub-games, so its per-side totals are not bounded
> by that band.
